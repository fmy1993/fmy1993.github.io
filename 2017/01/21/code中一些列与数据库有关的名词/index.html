Entity类以及一些用code操作数据库有关的名词

发表于 2017-01-21
  1. ADO(ACTIVEX DATA OBJECTS)应用层的数据访问接口
  2. ODBC 数据库驱动接口
  3. OLE DB 系统级数据访问接口
  4. DAO (DATA ACCESS OBJECTS) 对象的数据访问接口
  5. ADO是基于全新的OLE DB技术而设计的.是一种基于应用程序层的数据访问接口.它是DAO/RDO的后继产物.
  6. DAO是VB最先采用的面向对象的数据访问接口,通过DAO访问MICROSOFT JET数据库引擎.
  7. ADODC是封装ADO的VB数据控件,可减少代码编写.
  8. ADODB是数据对象,操作数据库需通过编写代码 来实现.
简而言之，就是用来实现一些特定功能的api.
  1. odbc/jdbc连接数据库，得到一个类似db的实例。
  2. 使用上述实例通过特定api（ado/dao,….）配合sql语句来实现对数据库的CRUD。
entity类创建用来对应数据库中的某张表（1vs1）,一般建立为final类（表结构没有继承关系），属性对应字段，实现get/set方法。

参考：http://blog.csdn.net/smxjant/article/details/52583124
O/R Mapping:
http://www.cnblogs.com/teddyma/archive/2010/08/01/1789714.html
简单映射 1. Class <-> Table 一个Class一般可以映射为一个Table，一个Class的实例对应Table的一行数据。但是，一个Table中的每行数据，一般都需要有一个主键来唯一标识这行数据，而一个Class的每个实例，则不一定需要一个唯一标识。 

            2. Property <-> Field  一个Class的Property一般可以直接映射为Table的一个Field。但是，他们的数据类型不一定直接匹配。如果他们代表的数据类型的语义上可转换，则Field的类型，应大于等于Property的数据类型。如果他们代表的类型语义上不可转换，则需要在应用程序层面，进行自定义的转换。

PS:final类的好处：
    1.不能被继承或者改变。
    2.线程安全
